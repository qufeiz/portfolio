01

The problem

A business runs on two kinds of knowledge that almost never live together: the documents — contracts, notices, scanned case files — and the structured data — the spreadsheets and tables of spend, vendors, renewals. Answering one real question often means opening both, reading by hand, and trusting that whoever summarized it didn't get a number wrong.

General chatbots make this worse: they answer fluently whether or not the source actually says so. For a business — especially one in a regulated domain — a confident wrong answer with no traceable source is not usable. The need wasn't a PDF chatbot; it was a grounded assistant where every fact is checkable.

02

What it does

Nucleus is an AI business assistant that answers a free-form question — in English or Hebrew — over both uploaded documents and structured data, and returns an answer with a citation on every fact, so you can always check where it came from.

  • Cited Q&A across sources — one reply can pull from a SQLite-backed table of contracts or maintenance spend and from a document corpus (e.g. a case file), each claim carrying its own citation token — a row id or a PDF page.
  • Upload anything, ask immediately — PDFs, scanned documents, Excel and CSV go to Google Gemini File Search, which reads scanned/image PDFs natively (no separate OCR step), and become query-able right away.
  • Access control — admins create accounts and can remove anyone instantly; each user sees only their own uploaded documents, scoped at retrieval time, not just hidden in the UI.
  • Honest refusal — when the data genuinely doesn't contain the answer (e.g. a "who's overdue?" question against a table with no payment-status field), it says so and names the columns that do exist, rather than inventing one.

A logged-in user asks; Nucleus routes to the right source(s), grounds the answer in the retrieved evidence, validates it, and shows the answer with its citations and the routing decision behind it.

03

How it works

Nucleus is a single Next.js application: the polished UI and the retrieval/answer engine run together in one deployment. Every question takes the same grounded path — route → retrieve → ground & cite → validate — and the engine surfaces its routing decision so the answer is traceable, not a black box.

01 One app A single Next.js (App Router) application on Vercel — the UI and the retrieval/answer engine run together in-process, same-origin (no second service to deploy). The whole thing is owned and deployed once.
02 Auth & access Supabase auth with a profiles table + role trigger (the first signup becomes admin). Each user sees only their OWN uploaded documents; an admin can provision accounts and deactivate ("kick out") anyone, who loses access on their next request.
03 Router A real model call decides which source(s) can answer a question — the structured (SQL) lane, the document lane, or both — and reports its decision and rationale, with a guard that keeps unrelated corpora from leaking into each other.
04 Document retrieval Google Gemini File Search — managed chunk / embed / retrieve-with-citations for uploaded PDFs, scanned documents, Excel and CSV. Each upload is tagged with its owner so retrieval is scoped per-user; grounding chunks map back to page-level citation tokens.
05 Answer model An OpenAI-compatible answer model selected entirely by env — DeepSeek by default in the demo, swappable to Gemini / Azure OpenAI / Vertex in one config change, no code change.
06 Validation gate A pure validateAnswer() gate runs before delivery: every citation must resolve to evidence retrieved this turn, every factual claim must be cited, and a stated number must actually be supported by the row or page it points at. A fluent-but-ungrounded answer is rejected, not shown.

The model layer is provider-neutral by design: the document lane is Gemini File Search, and the sentence-writing answer model is a one-line env swap between DeepSeek, Gemini, Azure OpenAI, or Vertex — so the same app can run on a free dev key for a demo or a compliant cloud key for real data.

/ Product screens

04

Outcome

Nucleus shipped as a working product: a single Next.js app, live on Vercel, with real Supabase auth and per-user document isolation, real Gemini File Search retrieval, a swappable answer model, and English + Hebrew support — every headline capability verified live against synthetic sample data, never a mock passed off as done.

It comes with a non-technical owner's handoff guide so the client can stand up their own copy — their own GitHub repo, their own Vercel, their own Supabase and keys — and, for real regulated data, swap the answer model to a compliant provider without a rebuild. The demo runs on sample data so the whole workflow can be walked end to end.

05

Role & context

A real client engagement, designed and delivered end to end as a shipped product — built solo: the engine, the auth and access control, the document and structured retrieval, the answer pipeline, the admin surfaces, and the owner handoff doc. The live demo runs on synthetic sample data; no real client records or identities are reproduced here. Demo logins are provided on request, not published.
