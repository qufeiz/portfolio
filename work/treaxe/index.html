01

The problem

Construction back-offices leak money in the gaps between tools: a bid wins but the project never gets set up cleanly, materials get ordered against the wrong budget, change orders go un-billed, invoices slip.

The value isn't any single feature — it's that the whole workflow is connected, so a signed estimate becomes a project becomes an invoice without anyone re-keying data or dropping a step.

02

What it is

TreAxe is a production construction-operations platform for small-to-mid contractors — the kind of crews who run the whole business out of a truck, a phone, and a pile of spreadsheets. It replaces that sprawl with one role-aware app that follows a job across its entire life.

That's roughly 42 screens covering CRM, estimating with line items and e-signatures, scheduling and time tracking, procurement, field documentation, invoicing, and a client-facing portal — all multi-tenant, all under row-level security.

03

How it works

The frontend is React + Vite + TypeScript with a layered architecture — pages → context → services → Supabase — so business logic stays out of components. Global state lives in typed context providers (auth/roles, all business entities, and English/Spanish i18n for bilingual crews). Every entity has its own CRUD service over the Supabase client.

The backend is Supabase Postgres with row-level security on every table, multi-tenant by org_id — a contractor only ever sees their own data, enforced at the database, not the UI. All external side-effects are isolated behind Supabase Edge Functions (Deno):

  • Payments — Stripe Checkout for one-time invoice payments (card / ACH) and org subscriptions, with a server-side invoice status state machine so a payment can't leave an invoice inconsistent.
  • Email — Gmail send via the user's stored OAuth token (auto-refreshing), used for estimates, change orders, and invoices, every send logged to an activity trail.
  • Material pricing — live producer-price-index lookups (24h cached) so estimates use real market rates.

It ships continuously: pushing to main auto-deploys to production (www.treaxe.io) via Vercel, gated by a Playwright suite — unit tests, end-to-end journeys against a live environment, and a post-deploy smoke check on the critical path.

04

Engineering depth

  • Security at the data layer — RLS policies, not just app-level checks; the multi-tenant boundary holds even if the UI is bypassed.
  • Hardened auth — the Supabase client uses an in-process lock instead of the browser Web Locks API, fixing a class of sign-in deadlocks that only appear on real (non-incognito) browsers — the kind of bug that passes every clean-profile test and still breaks production.
  • Honest tests — journey tests assert the real outcome (the row persisted, the email actually sent) rather than a green status pill, with no silent skips.
05
